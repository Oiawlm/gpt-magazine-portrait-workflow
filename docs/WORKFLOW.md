# Workflow

## 核心分工（重要）
本工作流采用明确的角色分工，请勿混淆各模型职责：
- **Codex 生图能力**：当前 Codex 会话自带的生图能力，要求使用者账号/组织具备图片生成权限和可用额度，不要求用户配置 OpenAI API Key；负责生成多视图参考图和最终杂志写真图片，视觉效果以 GPT Image 经验为基准
- **Doubao-Seed-2.0-Pro**：图片理解和提示词生成模型，不负责生图，仅负责读取人物图、风格图，生成提示词队列
- **Codex**：流程控制器和最终生图执行者，负责默认目录创建、文件管理、多视图生成、任务校验、最终图片生成、结果记录和仓库维护
- **Claude Code**：协作者，负责模型切换、长文本处理、模板生成、文档整理
- **DeepSeek V4 Pro**：当前不进入本工作流；不要作为文本整理、fallback 或可选步骤加入

## 目标

把一个人物的多角度参考图，转化为一组高质量杂志写真图片。最终目标是跑通“用户拖图 -> Codex 自动创建默认运行目录 -> Codex 生成多视图参考图 -> Doubao 生成任务队列 -> Codex 校验并最终生图 -> 结果记录”的闭环。

当前稳定 MVP 在 Codex Desktop 尚不能稳定暴露拖入附件路径时，使用 `assets/inbox/` 作为固定图片入口。用户把 3-5 张人物照片放入该目录后，Codex 直接运行 `start_character_run.ps1` 扫描该目录启动流程；图片文件名可以任意，只要是常见图片扩展名。如果当前环境能暴露拖图路径，也可以走 `-SourceImagePath` 路线。

## 阶段 1：人物资料准备

1. 当前稳定 MVP：用户把同一人物的多角度照片放入 `assets/inbox/`，并触发 `gpt-magazine-portrait` 工作流。
2. Codex 调用 `skills/gpt-magazine-portrait/scripts/start_character_run.ps1`，自动扫描 `assets/inbox/` 并创建目录：`assets/characters/<auto-character-id>/`。
3. 可选增强路线：如果当前 Codex 环境明确暴露拖入图片的本地路径，Codex 可将路径传给 `-SourceImagePath`。
4. 如果 `assets/inbox/` 没有图片，且当前 Codex 环境也不能读取拖入图片路径，流程停在输入读取阶段，明确说明缺少可用图片路径；不要声称能把对话图片自动保存到临时目录。
5. 原始照片默认保存到 `reference/originals/`。
6. 多视图参考图默认保存到 `reference/multiview/`。
7. 任务队列默认保存到 `tasks/`。
8. 最终图片默认保存到 `generated/`。
9. 运行 manifest 默认保存到 `runs/`。
10. 人物 Markdown 由 Codex 根据图片可观察信息先自动草拟，不要求用户手动填写。

## 阶段 1.5：生成多视图人物参考图

**本步骤由 Codex 生图能力完成，不是 Doubao-Seed-2.0-Pro 完成。**

1. 基于用户上传的多角度人物照片，使用 Codex 生图能力生成一张标准化的人物多视图参考图（character reference sheet）。
2. 提示词模板可参考：`templates/multiview_reference_prompt.template.md`。
3. 生成的多视图参考图必须遵循模板固定结构：纯白色背景、细黑色分区、上排 4 张全身站姿、下排 4 张坐姿/特写/身体细节、右侧列 3 张头部特写和 1 个人物信息区，用于锁定人物一致性。
4. 将生成的多视图参考图保存到 `assets/characters/<name>/reference/multiview/` 目录，作为后续所有生成任务的核心人物参考。
5. Doubao-Seed-2.0-Pro 的职责是读取这张多视图参考图和风格参考图，生成后续的提示词队列。
6. 原始照片横向拼版、截图拼版、手工拼接图，或只包含 4 个半身头像视角的简化 reference sheet，不算多视图参考图成功结果，不得作为后续 Doubao 读图输入。
7. 如果 Codex 生图工具返回服务器错误、超时或不可用，流程停在本阶段，明确说明“多视图参考图生成失败，稍后重试”；不要创建 fallback 拼版，不要继续进入提示词队列或最终写真。

本阶段必须区分三种结论：

| 结论 | 判定标准 | 下一步 |
|---|---|---|
| 启动链路成功 | 人物目录、`reference/originals/` 和 manifest 已生成 | 继续尝试多视图生图，但不能写成多视图成功 |
| 多视图生图成功 | `reference/multiview/` 中存在 Codex 生图能力生成的 AI 标准化 reference sheet | 进入阶段 4，交给 Doubao 生成提示词队列 |
| 多视图生图失败 | Codex 生图工具报错、超时、不可用，或只生成原图横向拼版 | 停止流程，记录失败原因，稍后重试 |

## 阶段 2：风格资产准备

1. 使用 `assets/style-reference/` 中的高质量风格参考图。
2. 先做图片清点，生成 manifest。
3. 再做质量筛选，排除风格极端、不可复用或不适合通用迁移的图片。
4. 对剩余图片做风格聚类。
5. 把聚类定稿为 style pack。

## 阶段 3：人物风格谱系蒸馏

1. 读取当前人物已有生成图。
2. 为每张图记录风格定位、视觉关键词、穿搭关键词、构图关键词、排版关键词。
3. 提炼人物适合的主线、补充线和惊喜线。
4. 后续任务队列必须同时参考外部 style pack 和人物风格谱系。

## 阶段 4：提示词队列生成

1. Claude Code 通过 CC Switch 选择 Doubao-Seed-2.0-Pro。
2. Doubao 读取人物参考图、风格参考图和人物谱系。
3. 生成结构化任务队列，任务至少包含：
   - `task_id`
   - `character`
   - `style_pack_id`
   - `reference_character_image`
   - `reference_style_images`
   - `core_identity_constraints`
   - `final_prompt_zh`
   - `negative_prompt_zh`
   - `output_filename`
   - `status`
   - `priority`
5. 任务队列文件统一输出到：`assets/characters/<人物名>/tasks/` 目录，命名建议为 `first_round_prompt_queue.json`、`second_round_prompt_queue.json` 等。
6. 第一轮默认生成 4 个任务；不要让用户输入张数，后续版本再支持自定义数量。
7. Codex 校验任务队列，只修正明显规则问题，不随意扩张任务数量。

## 阶段 5：自动执行前检查

用户拖图并触发工作流，本身视为授权执行当前 MVP 默认流程。不要在正常新人物运行中再询问人物名、目录、生成张数或是否开始。

Codex 只在以下情况停下来说明问题：

- 缺少 Codex 生图能力。
- Codex 生图工具服务器错误、超时或不可用，导致多视图参考图生成失败。
- 缺少 Claude Code + CC Switch 或等价 Doubao-Seed-2.0-Pro 接入。
- 任务队列 JSON 校验失败且无法自动修正。
- 人物参考图或风格参考图路径不存在。
- 输出文件会覆盖已有图片。

`check_workflow_prereqs.ps1` 只检查仓库本地文件、模板和队列校验脚本，不验证 Codex 生图能力、Doubao 接入或拖图路径暴露能力。

`assets/inbox/` 是当前稳定 MVP 的默认图片入口。拖图即跑仍是最终目标；只有当前 Codex 环境能暴露拖入附件路径时，才走纯拖图路线。

## 阶段 6：Codex 生图执行

1. 使用 Codex 生图能力生成最终杂志写真图片。
2. 第一轮默认执行 4 个任务；如任务队列少于 4 个，则执行全部任务。
3. 如果文字漂移但画面很好，优先让 Codex 做定向修正，不立刻重跑整批。
4. 保存图片到 `assets/characters/<name>/generated/` 或人物目录指定位置。
5. 验证文件存在、大小正常、图片可打开。
6. 仅使用 Codex 生图能力作为执行路线。

## 阶段 7：记录结果

1. 更新人物 Markdown。
2. 更新任务队列 JSON 和 Markdown。
3. 写入 `output-records/` 运行记录。
4. 当前 MVP 不要求用户逐张做“保留/淘汰/人物像不像”等复杂反馈。

## 阶段 8：后续优化（可选）

如果要继续优化，下一轮不应盲目扩图。先判断：

- 哪些风格成功。
- 哪些风格只适合作为备选。
- 哪些问题来自提示词，哪些问题来自模型文字控制。
- 是否真的需要继续生成。

只有当规则被吸收后，才进入下一轮任务队列。开源 MVP 可以先不执行本阶段。
## 提示词生成规范
### 基础结构
```
[人物核心特征] + [风格特征] + [构图与场景] + [光影与色彩] + [画质要求] + [负面提示]
```
### 各部分说明
1. **人物核心特征**（必填）：精确描述人物的五官、发型、体型、标志性特征，例如："中国男性，30岁，短发，方脸，高鼻梁，左眉有一道疤，穿黑色皮夹克"
2. **风格特征**（必填）：参考风格图的风格关键词，例如："90年代港式警匪电影海报风格，高对比度，冷色调，颗粒感"
3. **构图与场景**：描述画面构图、拍摄角度、背景场景，例如："中景构图，仰拍，夜晚城市街头背景，霓虹灯"
4. **光影与色彩**：描述光影效果和色彩倾向，例如："侧光，面部有硬朗阴影，整体蓝橙对比色调"
5. **画质要求**：统一添加："高清，8K，杂志级画质，细节丰富，无畸变"
6. **负面提示**（必填）：统一添加基础负面提示："卡通，动漫，插画，低分辨率，模糊，变形，扭曲，多余肢体，文字错误，水印"
## Codex 生图执行指南
### 操作步骤
1. 读取任务队列中的人物参考图、风格参考图、`final_prompt_zh` 和 `negative_prompt_zh`。
2. 按任务队列逐一生成。
3. 生成后检查人物特征是否符合要求，风格是否匹配。
4. 将图片保存到任务指定输出路径。
5. 命名保持与任务队列中的 `output_filename` 一致。
### 注意事项
- 第一轮默认 4 张；如果一次性执行状态不稳定，可由 Codex 内部分批处理，但不要让用户手动选择批次
- 如果文字漂移但画面整体符合要求，优先定向修正文字，不要重新生成整批
- 如果人物特征变形，调整提示词中人物描述的权重，把核心特征放在最前面
## 常见问题排障
| 问题 | 原因 | 解决方法 |
|------|------|----------|
| 人物特征变形/不像本人 | 参考图不足，或提示词中人物特征描述不精确 | 增加多角度参考图，在提示词最前面强调核心特征 |
| 风格与参考图不符 | 风格关键词描述不准确，或参考图特征不明显 | 重新提炼风格关键词，增加风格参考图数量 |
| 文字错误/漂移 | GPT Image 对文字控制能力有限 | 简化文字内容，或使用编辑功能单独修正文字 |
| 画面有多余元素/畸变 | 负面提示不足 | 补充负面提示词，明确排除不需要的元素 |
| 生成速度慢/失败 | 网络问题或服务器繁忙 | 等待几分钟后重试，或减少单次生成的任务数量 |
## Skill 脚本使用说明
### 创建人物目录
```powershell
powershell -ExecutionPolicy Bypass -File .\skills\gpt-magazine-portrait\scripts\make_character_dirs.ps1 -CharacterName "your_character_name"
```
### 验证任务队列
```powershell
powershell -ExecutionPolicy Bypass -File .\skills\gpt-magazine-portrait\scripts\validate_queue.ps1 -QueuePath "path/to/your/task_queue.json"
```
